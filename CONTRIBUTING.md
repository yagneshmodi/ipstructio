## Ways to Contribute

You can contribute in many ways, including:

* Reporting bugs
* Suggesting new features or improvements
* Fixing issues
* Improving documentation
* Refactoring code
* Improving UI/UX
* Writing tests

---

## Reporting Bugs

If you find a bug:

1. Check existing issues to see if it has already been reported.
2. Open a new issue with the following details:

   * Clear and descriptive title
   * Steps to reproduce the issue
   * Expected behavior
   * Actual behavior
   * Screenshots or logs (if applicable)
   * Environment details (OS, browser, version)

---

## Feature Requests

Feature requests are welcome!

When opening a feature request, please include:

* The problem you are trying to solve
* Proposed solution or idea
* Why this feature would be useful
* Any alternatives you considered

---

## Development Setup

> *Setup instructions may evolve as the project matures.*

General steps:

1. Fork the repository
2. Clone your fork:

   ```bash
   git clone https://github.com/yagneshmodi/IPStructio.git
   ```
3. Create a new branch:

   ```bash
   git checkout -b feat/your-feature-name
   ```
4. Install dependencies
5. Start the development server
6. Make your changes

---

## Testing

* Ensure your changes do not break existing functionality
* Add or update tests where applicable
* Run the test suite before submitting a PR

---

## Code Style & Guidelines

* Follow the existing coding style and project structure
* Keep commits small and focused
* Write clear and descriptive commit messages
* Avoid unrelated changes in a single pull request
* Comment complex logic where necessary

---

## Submitting a Pull Request

1. Push your changes to your fork:

   ```bash
   git push origin feature/your-feature-name
   ```
2. Open a Pull Request against the `main` branch
3. Fill out the PR template completely
4. Link any related issues
5. Be responsive to feedback and review comments

---

## Pull Request Review Process

* Maintainers will review PRs as time permits
* You may be asked to make changes before approval
* Once approved, a maintainer will merge the PR

---

## Design Principles

When contributing, keep these principles in mind:

* **Simplicity over complexity**
* **Clear UI and usability first**
* **Security and correctness**
* **Auditability and traceability**
* **Extensibility**

---

## Security Issues

If you discover a security vulnerability, **do not open a public issue**.
Instead, report it privately to the maintainers.
